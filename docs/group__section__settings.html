The LSB Settings

The LSB Settings dialog provides users with the ability to configure a wide range of behaviors and preferences within the LSB application. All available settings are grouped into four major categories — General, Colors & Fonts, Shells, and Views — each covering a different aspect of the application’s functionality. The following sections describe these categories in detail, explaining the purpose of each option and illustrating how individual settings influence the overall operation and appearance of the LSB application.

General category

The General category contains the core application‑wide settings for the LSB environment. This section provides access to several fundamental configuration options that influence how the LSB application behaves during everyday use. The settings included in this category are as follows, and each one plays a role in shaping the overall user experience.

  • Aggregation time window for multiple input parameter changes — This setting specifies the duration, in milliseconds, of a time window used to aggregate multiple input‑parameter updates for a single LSB view control. A control may define several input parameters, and in many scenarios multiple parameters may change at nearly the same moment or within a very short interval. If the aggregation window is set to 0 milliseconds, every individual parameter change immediately triggers its own data load and content refresh, which is typically undesirable because these updates should be consolidated into a single operation. By assigning a small non‑zero duration, all parameter changes that occur within that window are treated as a single combined update, resulting in only one data load and one content refresh. It is important to note that this window operates as a moving time window. For example, if the window length is D and the first parameter change occurs at T₀, and a second change occurs at T₁ where (T₁ − T₀) ≤ D, both changes are merged into one action, and the actual data load/refresh is scheduled for T₁ + D. If a third change occurs at T₂ where (T₂ − T₁) ≤ D, the refresh is postponed again to T₂ + D. This deferral continues until no further parameter changes occur within D milliseconds after the most recent change. In general, if n parameter updates occur close together (each pair separated by less than D), the final data load and refresh will occur at Tₙ + D.
  • Default view size — This option defines the initial window dimensions applied when an LSB view is opened. Every newly opened view begins with this default size unless explicitly overridden elsewhere.
  • Select the icon theme for the application — This setting determines which icon theme the LSB application uses. All available themes are presented in a selectable list, allowing the user to choose the preferred visual style. After selecting a new theme, the application must be restarted for the change to take effect. The current default theme is default.
  • Select the language for the application - This setting sets the laguange to be used for the application UI. There is no other languages currently available for LSB, but some languages will be added soon.
  • Show the GIT client window during execution for enlistment or refresh — The LSB application relies on Git to synchronize the default view repository, the help‑documentation repository, and any additional user‑configured repositories. During synchronization, the application invokes Git commands to update local files. This setting controls whether a console window is displayed while these Git operations are running.
  • Enable proxy server for Web requests at port — This experimental feature enables an internal proxy server designed to route WebView requests that originate from local content (which typically has a null origin) to remote web endpoints, thereby mitigating CORS restrictions. Without this proxy, local pages are unable to access responses from remote servers unless those servers explicitly allow CORS requests — something that is uncommon in many environments. Enabling the proxy allows these local‑origin requests to succeed by transparently forwarding them through the proxy server.

Color & Fonts category

The Colors & Fonts category includes all configuration options related to the visual appearance of the LSB application. Within this section, users can adjust the color schemes and font settings that determine how text, controls, and interface elements are rendered throughout the environment, allowing the overall look and readability of LSB to be customized to individual preferences.

  • Active view caption — These settings allow users to customize both the colors and the font used for the captions of active view controls. Users may click the foreground or background color fields to modify them, and use the “Choose…” button to select a different font for the active control’s caption. The Preview panel immediately reflects any adjustments, allowing users to see the final appearance of their changes in real time.
  • Inactive view caption — These settings provide similar customization options for the captions of inactive view controls. Users can modify the foreground and background colors, as well as choose a different font by clicking the “Choose…” button. As with the active caption settings, the Preview panel updates instantly to show the resulting appearance of any changes made.
  • Shell window content — This setting controls the colors and font used when displaying shell‑related windows, such as the Git console window that appears during view‑repository synchronization. Any modifications to these appearance settings are shown immediately in the Preview pane, ensuring users can verify the visual effect before applying the configuration.

Shells category

The Shells category enables users to configure the command‑line environments that can be used within LSB views. This settings page includes a list of all registered shells, along with a set of tool buttons that allow users to add, remove, or modify entries in the shell list. These controls make it easy to manage which shells are available for executing scripts inside LSB views.

  • Scan - This button scans the host operating system for all available shells. Any shells that are detected are automatically added to the shell list using their default configuration settings.
  • Edit - This button becomes enabled only when a shell entry in the list is selected. Clicking it opens the shell‑setting editor dialog, allowing the user to modify the configuration of the selected shell. For details on customizing shell settings, refer to the “Edit settings for xxx” section below.
  • Add - This button creates a new shell entry in the shell list. It launches the shell‑setting editor dialog with empty fields so the user can define the configuration for the new shell.
  • Delete - This button removes the currently selected shell from the shell list. It is enabled only when a shell entry is actively selected.
  • Clear - This button removes all shell entries from the shell list at once.

Note: Double‑clicking any item in the shell list performs the same action as clicking the Edit button; it opens the shell‑setting editor dialog so the user can update that shell’s configuration.

Edit settings for xxx

This dialog is used to create and modify the configuration settings for a shell that will be available within the LSB application. An LSB view can execute shell scripts only when a corresponding shell definition has been configured, and this dialog is the primary interface through which those shell settings are established. In most cases, any shell intended for use in LSB views is first registered and customized through this editor.

Here are the configuration options that can be defined for a shell:

  • Name — The display name of the shell. This value also serves as the shell type referenced within shell scripts embedded in LSB views.
  • Path — The absolute filesystem path to the shell’s executable program.
  • Reference / Module list — A list of shell‑script files that will be automatically loaded when the shell process is created. The contents of these files are executed before any user‑defined scripts in the LSB view, allowing those scripts to rely on variables, helper functions, or utilities defined in the referenced modules.
  • Placeholders — A set of placeholder tokens used within the script wrapper. Before execution, these placeholders are replaced with actual runtime values. They support the execution of the user‑defined script inside the shell process and help return data back to the LSB application.
    • Customized user script — The placeholder representing the user‑defined shell script embedded in the LSB view file.
    • Intermediate data file — A placeholder that resolves to the full path of the temporary data file used to store output generated by the user‑defined script. After execution, the LSB application reads this file and uses the extracted data within the view.
    • First half of signature — A placeholder for the first portion of a randomly generated signature value. This signature is used during script execution to detect completion and validate output. See The Wrapper Script for Shells for more details.
    • Second half of signature — A placeholder for the second portion of the signature. Combined with the first half, it forms the complete signature used to mark the end of script execution. See The Wrapper Script for Shells for additional explanation.
    • Path to view file — A placeholder that resolves to the full path of the currently opened LSB view file.
  • User script wrapper — A block of shell script that wraps the user‑defined script before execution. The wrapper is responsible for running the user script, writing its output to the intermediate data file, and emitting the signature data to the shell console to signal completion. Additional helper logic may also be included to support script execution.

  • Script execution timeout — The default timeout applied to user‑defined script execution. Individual scripts may override this value through their ScriptData configuration in the LSB view file, see ScriptData. If no custom timeout is specified there, this default timeout is used.

Views category

The Views category allows users to configure the set of view repositories that the LSB application relies on. This settings page includes a list of all registered view repositories, along with a collection of tool buttons that make it easy to add, remove, or modify repository entries. Through these controls, users can manage where LSB searches for view files and how those repositories are organized within the application.

  • Edit — This button becomes available only when an LSB view repository is selected in the repository list. Clicking it opens the “Edit view repository settings for xxx” dialog, where the user can modify the configuration of the selected repository. Details on customizing repository settings are provided in the section below.
  • Synchronize — This button is enabled when the selected LSB view repository is a Git‑based repository and a valid remote Git URL has been configured. Activating the button initiates a synchronization process that updates the local enlistment with the latest view files from the remote Git repository.
  • Add — This button allows the user to create a new LSB view repository entry. It opens the “Edit view repository settings for xxx” dialog with empty fields so the user can define the settings for the new repository.
  • Delete — This button removes the currently selected LSB view repository from the list. It is enabled only when a repository entry is actively selected.
  • Clear — This button deletes all entries from the view repository list, effectively resetting the list to an empty state.

Edit view repository settings for xxx

When the user chooses to add a new LSB view repository or modify an existing one, the application opens this editor dialog to allow full customization of that repository’s settings. This dialog serves as the central interface for defining how the LSB application locates, accesses, and synchronizes the view files associated with the selected repository.

  • Name — The display name of the LSB view repository. This value appears in the Repository column of the view list shown in the “Open a view” dialog, helping users identify which repository each view originates from.
  • Local view root — The local filesystem path that points to the root directory of the repository. All LSB view files stored under this folder are treated as part of the repository.
  • GIT repository URL — If the repository is synchronized from a remote Git source, this field specifies the URL of that remote Git repository. When this value is left empty, the repository is treated as a purely local repository, and no synchronization operations will be performed for it.
  • Refresh interval — This setting applies only when the repository is linked to a remote Git source. It defines how frequently the LSB application should synchronize the local enlistment with the remote Git repository. Automatic synchronization occurs only during application startup and follows this interval. Manual synchronization — triggered by clicking the Synchronize button in the Views configuration page — ignores this interval and performs the update immediately.
  • The repository is enabled to use the views hosted in it — This toggle determines whether the LSB views contained in the repository should appear in the “Open a view” dialog. If the option is unchecked, the view files remain on the user’s machine but are intentionally hidden from the view list, effectively disabling the repository without removing it.
